# assh Release Audit Design

Date: 2026-05-15
Status: Awaiting user review

## Context

`assh` is a Go CLI shipped through two release surfaces:

- GitHub Releases provide native binaries built by GoReleaser.
- The npm package `agent-assh` installs a JavaScript wrapper that downloads the matching release binary during `postinstall`.

The repository already has CI for Go formatting, vetting, tests, race tests, Go linting, Markdown linting, GoReleaser config checks, npm smoke tests, and npm package dry runs. The release workflow runs on tags matching `v*`, verifies that `package.json` version matches the tag, builds release artifacts, and publishes the npm package.

The goal is to review whether that flow is actually release-ready and to fix concrete issues found by local verification.

## Goals

- Run the same practical checks expected before a release.
- Verify that tag-driven release behavior matches package and installer assumptions.
- Check that GoReleaser archive names, checksum names, platform mapping, and npm install script agree with each other.
- Check that the npm package contains the files needed to run the installed wrapper.
- Fix any concrete lint, test, packaging, or release-flow defects found during the audit.
- Keep changes focused on release readiness.

## Non-Goals

- Do not redesign SSH command behavior unless a test or release check exposes a defect.
- Do not add unrelated product features.
- Do not rewrite the module structure.
- Do not perform real publishing to GitHub Releases or npm.
- Do not change package identity unless the audit proves the current identity cannot work.

## Audit Scope

The audit covers these files and boundaries:

- `.github/workflows/ci.yml`
- `.github/workflows/release.yml`
- `.goreleaser.yaml`
- `package.json`
- `bin/assh.js`
- `scripts/install.js`
- `scripts/platform.js`
- `scripts/smoke-test.js`
- Go source and tests under `cmd/` and `internal/`
- release-facing documentation that is linted or packaged

## Verification Commands

Run the relevant local equivalents of CI and release checks:

- `test -z "$(gofmt -l .)"`
- `go vet ./...`
- `go test ./...`
- `go test -race ./...`
- `golangci-lint run` when available locally
- Markdown lint when available locally
- `goreleaser check` when available locally
- a GoReleaser snapshot or dry-run check when feasible without publishing
- `npm run smoke`
- `npm pack --dry-run`

If an optional tool is not installed locally, record that limitation and verify the closest available substitute.

## Release Flow Checks

Review the tag-based release flow for these properties:

- Release runs only from intended tags.
- The tag name and `package.json` version must match.
- GoReleaser builds the OS and architecture pairs expected by the npm installer.
- Archive names generated by GoReleaser match the names requested by `scripts/install.js`.
- `checksums.txt` is present and the installer verifies SHA-256 before copying the binary.
- The npm package includes every file required by `bin/assh.js`, `scripts/install.js`, and `scripts/platform.js`.
- The package smoke test does not depend on network access or real releases.
- CI checks enough of the release path to catch broken packaging before a tag is pushed.

## Fix Policy

Fixes should be minimal and directly tied to a failed command, a broken release assumption, or a clear packaging mismatch. Acceptable fixes include:

- CI or release workflow corrections.
- GoReleaser configuration corrections.
- npm package metadata or file list corrections.
- installer, wrapper, or platform mapping corrections.
- focused tests that lock the corrected release behavior.
- documentation corrections when docs contradict the shipped release flow.

If the audit finds a larger design issue, document it as a risk and avoid broad refactoring in this pass.

## Output

The implementation pass should produce:

- a concise audit summary with findings and fixes,
- changed files, if fixes are needed,
- verification commands run and their results,
- explicit notes for any checks that could not run locally.
